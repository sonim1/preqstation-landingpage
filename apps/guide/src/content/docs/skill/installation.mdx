`preqstation-skill` is the core agent-side package for PREQSTATION. It owns the `preqstation` skill name, documents the lifecycle rules, includes optional shell-helper and legacy stdio bridge support, and also carries the local Claude plugin skeleton plus the dispatcher channel files used by the Claude Code dispatch path.

## Prerequisites

- Node.js 22+ installed
- A running PREQSTATION deployment at `https://<your-domain>`
- Access to the PREQSTATION web app if you want to use the recommended remote MCP login flow
- A PREQSTATION API token only if you plan to use shell-helper mode or the legacy local stdio bridge

## Install Skill (Recommended Per Agent)

Install the skill only for the agents that should own the core `preqstation` skill:

```bash
npx skills add sonim1/preqstation-skill -g -a claude-code
npx skills add sonim1/preqstation-skill -g -a codex
npx skills add sonim1/preqstation-skill -g -a gemini-cli
```

Avoid a bare shared install if you do not want OpenClaw-linked skills exposed through the global skills store.

PREQ engine values match these agent targets:

```bash
claude-code
codex
gemini-cli
```

## Dispatcher Choice Model

PREQSTATION separates **worker installation**, **dispatcher target**, and **engine selection**:

- Workers use the remote `/mcp` server and call PREQ tools directly.
- The web app queues one of two `dispatch_target` values when you use a dispatch action:
  - `claude-code-channel` via **Send to Claude Code**
  - `telegram` via **Send to Telegram** or copied Telegram/OpenClaw commands
- The task `engine` stays separate from the dispatcher target and decides whether the dispatcher launches `claude-code`, `codex`, or `gemini-cli`.

If you want the OpenClaw dispatcher path, set up Telegram plus `preqstation-openclaw`. This page focuses on the worker setup shared by all agents and the Claude Code dispatcher channel hosted in `preqstation-skill`.

## Recommended MCP Setup (HTTP + OAuth)

The recommended MCP runtime now lives in the PREQSTATION service itself at `/mcp`. Claude Code and Codex should connect to that remote endpoint and complete the browser OAuth flow when prompted.

## Claude Code

Register the remote PREQSTATION MCP endpoint:

<Tabs>
<TabItem label="Install">
```bash
claude mcp add --transport http preqstation https://<your-domain>/mcp
```
</TabItem>
<TabItem label="Update">
If already registered, remove and re-add:

```bash
claude mcp remove preqstation
```

Then run the install command above.
</TabItem>
<TabItem label="Verify">
```bash
claude mcp list
claude mcp get preqstation
```
</TabItem>
</Tabs>

### Optional: local Claude plugin and dispatcher channel

If you want the **Claude Code dispatcher** from the PREQSTATION web app, load the local plugin skeleton and start the local dispatcher channel runtime:

```bash
claude --plugin-dir /ABSOLUTE/PATH/TO/preqstation-skill
claude --mcp-config /ABSOLUTE/PATH/TO/preqstation-skill/.mcp.json --dangerously-load-development-channels server:preq-dispatch-channel
```

The dispatcher host itself runs inside Claude Code, but the queued task still keeps its own `engine`. A task sent through **Send to Claude Code** can still launch `codex` or `gemini-cli` as long as those CLIs are installed on the same machine.

</TabItem>
<TabItem label="Update">

Codex does not need a Claude-style local plugin or channel runtime for normal worker usage. If you want queued dispatch from the web app through the Claude Code dispatcher, keep Codex installed locally and select `Codex` as the task engine in the UI.

## When Authentication Starts

- Codex often probes the server during `add`, so the browser login flow may start immediately.
- Claude Code usually saves the server first and may show `Needs authentication` until first use.
- In Claude Code, authentication commonly starts when you first use the server or run `claude mcp get preqstation`.

## Dispatch Routing Summary

- **Send to Claude Code** queues `dispatch_target=claude-code-channel`.
- **Send to Telegram** queues `dispatch_target=telegram`.
- **Engine** remains a separate field and can still be `claude-code`, `codex`, or `gemini-cli` on either dispatcher path.
- **Worker-only usage** still works without any queued dispatch if you only want PREQ MCP tools inside the agent.

## Gemini CLI

Gemini CLI does not use the documented remote MCP registration flow here. Install the `preqstation` skill for `gemini-cli`, then use shell-helper mode if you need direct REST access. If you want queued dispatch through the Claude Code dispatcher, make sure `gemini` is also installed on that dispatcher machine:

```bash
export PREQSTATION_API_URL="https://your-preqstation-domain.vercel.app"
export PREQSTATION_TOKEN="preq_xxxxxxxxxxxxxxxxx"
export PREQSTATION_ENGINE="gemini-cli"  # optional fallback
```

## Shell Helper or Legacy Local Bridge (Optional)

If MCP is unavailable, you can still use the shell helper or the legacy local stdio bridge.

Set these variables first:

```bash
export PREQSTATION_API_URL="https://your-preqstation-domain.vercel.app"
export PREQSTATION_TOKEN="preq_xxxxxxxxxxxxxxxxx"
export PREQSTATION_ENGINE="claude-code"  # or codex, gemini-cli
```

## JSON Configuration Example

If you must manually configure the legacy local stdio bridge in settings:

```json
{
  "mcpServers": {
    "preqstation": {
      "command": "node",
      "args": ["/ABSOLUTE/PATH/TO/preqstation-skill/scripts/preqstation-mcp-server.mjs"],
      "env": {
        "PREQSTATION_API_URL": "https://your-preqstation-domain.vercel.app",
        "PREQSTATION_TOKEN": "preq_xxxxxxxxxxxxxxxxx"
      }
    }
  }
}
```

<Aside>
Replace `/ABSOLUTE/PATH/TO/preqstation-skill` with the actual installation path. Relative paths will not work. The recommended remote `/mcp` setup above does not need a local script path.
</Aside>

## Troubleshooting

**Remote MCP shows `Needs authentication`:**
- Confirm the server URL ends with `/mcp`
- Run `claude mcp get preqstation` or `codex mcp get preqstation` to trigger the first real connection
- Complete the browser login flow when prompted

**Legacy local bridge fails to start:**
- Verify Node.js is installed: `node --version`
- Check the script path is absolute
- Confirm `PREQSTATION_API_URL` is valid (should start with `https://`)

**Shell helper or legacy bridge returns `401 Unauthorized`:**
- Verify token format (should start with `preq_`)
- Test connectivity: `curl -H "Authorization: Bearer $PREQSTATION_TOKEN" "$PREQSTATION_API_URL/api/health"`

**Token rejected:**
- Generate a new token from the PREQSTATION web app `/api-keys` page
- Ensure token has not expired
