PREQSTATION의 태스크는 두 축으로 관리됩니다.

- **워크플로우 상태** — 카드가 현재 어떤 단계에 있는가
- **Run state** — 지금 이 순간 에이전트가 실행 권한을 요청했거나 점유했는가

이 분리가 이전의 `in_progress` / `review` 모델과 가장 큰 차이입니다.

## 워크플로우 상태

| 상태 | 의미 | 다음 액션 |
|---|---|---|
| **Inbox** | 새 태스크, 아직 계획 전 | 에이전트가 코드를 읽고 `preq_plan_task()` 실행 |
| **Todo** | 계획 완료, 구현 가능 | 에이전트가 `preq_start_task()`로 claim 후 작업 시작 |
| **Hold** | 막혔거나 일시정지 | 원인 해결 후 재개하거나 그대로 유지 |
| **Ready** | 구현 완료, 검증 대기 | 에이전트 또는 owner가 검증 후 `preq_review_task()` 실행 |
| **Done** | 검증 완료 | 수동으로 재오픈하지 않는 한 종료 |
| **Archived** | 활성 보드에서 숨김 | 수동 archive 전용 |

## Run state 오버레이

Run state는 워크플로우 상태와 별도로 추적됩니다.

| Run State | 의미 |
|---|---|
| `null` | 활성 실행 점유 없음 |
| `queued` | 요청은 됐지만 아직 에이전트가 claim 전 |
| `working` | 에이전트가 `preq_start_task()`로 claim 완료 |

OpenClaw나 Telegram dispatch가 먼저 `queued`를 설정할 수 있고, 이후 `preq_start_task()`가 이를 `working`으로 올립니다.

## 라이프사이클 브랜치

에이전트는 태스크를 한 번 읽고, 현재 상태에 맞는 브랜치 하나만 수행합니다.

### Inbox 에서

```text
preq_get_task(PROJ-284)         -> status=inbox
preq_start_task(PROJ-284)       -> run_state=working
로컬 코드 확인 + 계획 작성
preq_plan_task(PROJ-284, ...)   -> status=todo, run_state=null
stop
```

### Todo 에서

```text
preq_get_task(PROJ-284)           -> status=todo
preq_start_task(PROJ-284)         -> run_state=working
구현 + 테스트 + 배포
preq_complete_task(PROJ-284, ...) -> status=ready, run_state=null
stop
```

### Hold 에서

```text
preq_get_task(PROJ-284)         -> status=hold
preq_start_task(PROJ-284)       -> run_state=working
막힌 원인 조사 / 작업 재개
해결됨: preq_complete_task(PROJ-284, ...) -> ready
여전히 막힘: preq_block_task(PROJ-284, ...) -> hold
stop
```

### Ready 에서

```text
preq_get_task(PROJ-284)         -> status=ready
preq_start_task(PROJ-284)       -> run_state=working
검증 실행 (tests, build, lint)
성공: preq_review_task(PROJ-284) -> done
실패: preq_block_task(PROJ-284, ...) -> hold
stop
```

<Aside>
`preq_start_task()`는 실행 claim 액션입니다. 더 이상 `in_progress` 같은 별도 워크플로우 컬럼을 만들지 않습니다.
</Aside>
