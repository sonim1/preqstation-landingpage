PREQSTATION uses session cookies for the web UI, OAuth-backed MCP connections for Claude Code, Codex, and Gemini CLI, and bearer tokens only for shell-helper compatibility or direct REST automation.

## OAuth-Backed MCP

**For Claude Code, Codex, and Gemini CLI.**

Register the remote `/mcp` endpoint and complete browser OAuth:

```bash
# Claude Code
claude mcp add -s user --transport http preqstation https://your-domain.com/mcp

# Codex
codex mcp add preqstation --url https://your-domain.com/mcp

# Gemini CLI
gemini mcp add --scope user --transport http preqstation https://your-domain.com/mcp
```

OAuth usually starts on the first real request. Manage connected clients from `/connections`.

## Bearer Token Authentication

**For shell-helper compatibility and direct REST automation.**

### Token Format

```
Authorization: Bearer preq_a7f3c9e2b1d4f6a8c0e2b4d6f8a0c2e4
```

- **Prefix**: `preq_`
- **Length**: 32 random characters
- **Storage**: SHA-256 hashed in database
- **Type**: Stateless (no session required)

### Obtaining a Token

Legacy bearer tokens are retained for compatibility. Prefer `/mcp` OAuth for new installs, and keep using an existing token only when shell-helper mode or direct REST automation still depends on it.

### Using a Token

Pass the token in the `Authorization` header:

```bash
curl -H "Authorization: Bearer preq_xxxxx" \
  https://your-domain.com/api/tasks
```

### Token Expiration

Tokens can have optional expiration dates:

- **No expiration** — Token valid indefinitely (rotate manually)
- **With expiration** — Token invalid after specified date
- **Expired** — Returns `401 Unauthorized`

Check the legacy token state from your deployment's compatibility flow before reusing it.

## Session Cookie Authentication

**For web UI (automatic).**

### Session Cookies

Created when you log in via the web UI:

```
Set-Cookie: session=...HMAC-signed...;
  HttpOnly;
  SameSite=Strict;
  Secure;
  Path=/;
  Max-Age=86400
```

### Cookie Flags

- **HttpOnly** — Cannot be accessed by JavaScript (XSS protection)
- **SameSite=Strict** — No cross-site requests (CSRF protection)
- **Secure** — HTTPS only (man-in-the-middle protection)
- **Max-Age** — Configurable session timeout (default: 24 hours)

### Automatic Renewal

Sessions are automatically renewed on each request (up to Max-Age).

## Error Responses

### 401 Unauthorized

Token is missing, invalid, or expired:

```json
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Missing or invalid authorization header"
  }
}
```

### 403 Forbidden

Token is valid but scoped to different owner (single-owner system):

```json
{
  "error": {
    "code": "FORBIDDEN",
    "message": "Insufficient permissions"
  }
}
```

## Best Practices

### 1. **Never Hardcode Tokens**

❌ Bad:
```javascript
const token = "preq_xxxxx"; // Hardcoded!
```

✅ Good:
```javascript
const token = process.env.PREQSTATION_TOKEN;
```

### 2. **Use Environment Variables**

```bash
export PREQSTATION_API_URL=https://your-domain.com
export PREQSTATION_TOKEN=preq_xxxxx
```

### 3. **Rotate Tokens Regularly**

- Issue a new token
- Update agent configuration
- Revoke the old token
- Frequency: every 90 days minimum

### 4. **Separate Tokens by Environment**

- `local-dev` for development
- `ci-test` for CI/testing
- `prod` for production

### 5. **Monitor Token Usage**

Check audit logs for unexpected API calls.

### 6. **Revoke Immediately if Exposed**

If a token is compromised:
1. Go to the legacy token management flow provided by your deployment
2. Delete the exposed token
3. Issue a new one
4. Update your agents

Token revocation is immediate. Any agent using a revoked token will receive `401 Unauthorized` on the next API call.
