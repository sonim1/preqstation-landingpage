Remote `/mcp` plus browser OAuth is the default install path for Claude Code and Codex. This page is the legacy compatibility path for shell-helper mode and direct REST automation. Visiting `/api-keys` redirects to `/connections?legacy=1`.

API tokens allow coding agents to authenticate to the PREQSTATION REST API and perform task operations without OAuth.

## Token Format

- **Prefix**: `preq_`
- **Storage**: SHA-256 hashed in database (never stored in plaintext)
- **Length**: 32 random characters
- **Example**: `preq_a7f3c9e2b1d4f6a8c0e2b4d6f8a0c2e4`

## Current Status

Legacy API keys remain available only as a compatibility path. The current web app redirects `/api-keys` to `/connections?legacy=1` to point new installs back toward OAuth-backed MCP.

Use a bearer token only when you already maintain a shell-helper or direct REST setup that still depends on it.

## Using a Token

Use tokens when your client needs shell-helper compatibility or direct REST access.

### Recommended Remote MCP Setup

```bash
# Claude Code
claude mcp add -s user --transport http preqstation https://your-domain.com/mcp

# Codex
codex mcp add preqstation --url https://your-domain.com/mcp
```

OAuth usually starts on the first real request. Codex often triggers it during `add`, while Claude Code commonly starts it when you first use the server or run `claude mcp get preqstation`.

### Token-Based Shell Helper

Configure your environment like this:

```bash
export PREQSTATION_API_URL=https://your-domain.com
export PREQSTATION_TOKEN=preq_xxxxxxxxxxxxx
```

If you are using the shell helper, the PREQSTATION service still needs a remote URL and token.

## Revoking a Token

1. Open the legacy token management flow provided by your deployment
2. Revoke the token you no longer want to trust
3. Update any shell-helper or direct REST runtime that still references it

Revoked tokens are immediately invalid. Any agent using a revoked token will receive `401 Unauthorized`.

## Token Rotation

Best practice: rotate tokens periodically.

1. Issue a new token with the new name
2. Update your agent configuration with the new token
3. Revoke the old token

## Scoping

All tokens are scoped to your owner account. Tokens can:

- Read and create tasks
- Update task status and results
- List projects and project settings
- Create work logs

Tokens cannot:

- Access other user accounts (single-owner system)
- Modify user settings or issue new tokens
- Access audit logs or security events

## Security Notes

- **Never commit tokens to version control** — use environment variables or secrets management
- **Rotate tokens before sharing credentials** — if a token is exposed, revoke and issue a new one
- **Token prefix** (`preq_`) helps identify tokens in logs and error messages
- **Expiration** (optional) enforces automatic token invalidation after a set time
